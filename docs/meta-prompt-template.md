## 메타 프롬프트

```
당신은 솔로 개발자를 위한 PRD 전문가입니다.
아래 프로젝트 정보를 바탕으로 실용적이고 개발 가능한 PRD 문서를 작성해주세요.

## 프로젝트 개요

**프로젝트명**: OathKeeper: B2B AI 개발 사업 Deal Go/No-Go 의사결정 지원 에이전트

**핵심 목표**:
OathKeeper는 B2B AI 개발 서비스 회사에서 세일즈 담당이 가져오는 고객사 개발 요청 사업 Deal의 수행 여부를 체계적으로 분석하고, Go / No-Go / 조건부 판단을 지원하는 AI 에이전트 서비스이다.
현재 구두 보고 기반의 비구조적 의사결정 프로세스를 데이터 기반의 일관된 평가 체계로 전환하여, 의사결정 속도 향상, 리스크 사전 식별, 구성원 간 판단 기준의 일관성을 확보하는 것이 목표이다.


**사용자 시나리오**:
0. 사전 준비: 의사결정의 필요한 데이터를 구축
- 기 프로젝트 수행 사례
- 회사 정보: 사업 방향(사업 영역, 범위 등), 비즈니스 전략(단기/중기/장기 전략), deal 선정 기준, 사업 규정(인건비 단가, HW/SW/라이센스 비용, 인력 현황)

1. 영업 담당자가 Deal 정보를 노션에 입력(회의록, 메모 등 비구조화 텍스트)

2. 경영진(또는 의사 결정권자, 사용자)가 OathKeeper에 분석 요청: 이때 문서(word/pdf)나 요청 텍스트가 추가될 수 있음

3.1 OathKeeper 에이전트가 Deal 정보와 경영진(사용자) 입력 정보를 파싱하여 구조화된 형태로 요약, 정리
- 출력항목 예시
| 필드명 | 설명 | 입력 유형 |
|---|---|---|
| 고객사 정보 | 회사명, 규모, 산업군, 담당자 | 텍스트 / 선택 |
| 프로젝트 개요 | 요청 배경, 목적, 기대 효과 | 텍스트 (자유 입력) |
| 기술 요구사항 | 필요 기술 스택, 모델, 데이터 환경 | 텍스트 / 문서 업로드 |
| 예상 금액 및 납기 | 계약 금액, 요구 납기, 결제 조건 | 숫자 / 날짜 선택 |
| 특이사항 | 고객 특별 요구, 제약 조건, 보안 요건 등 | 텍스트 (자유 입력) |

3.2 OathKeeper 에이전트가 표준 평가 기준에 따라 Deal을 자동 스코어링하고, 종합 판단을 제시. 이때 사전 준비 단계에서 설정한 회사 사업방향, 전략 문서도 함께 고려함.
- 평가 기준 및 가중치(예시)

| # | 평가 기준 | 가중치 | 평가 항목 |
|:---:|---|:---:|---|
| 1 | 기술 적합성 | 20% | 자사 기술 스택으로 구현 가능한가? 추가 기술 학습 필요성은? |
| 2 | 리소스 가용성 | 15% | 현재 투입 가능한 인력/장비가 있는가? 타 프로젝트와 충돌은? |
| 3 | 수익성 | 20% | 예상 매출 대비 투입 비용이 적절한가? 목표 마진을 충족하는가? |
| 4 | 납기 리스크 | 15% | 고객 요구 일정이 현실적인가? 버퍼 기간이 충분한가? |
| 5 | 요구사항 명확성 | 10% | 고객의 요구사항이 구체적인가? 범위 변경 리스크는? |
| 6 | 전략적 가치 | 10% | 레퍼런스 확보, 신규 산업 진출, 기술 축적 등 장기적 가치 |
| 7 | 고객 리스크 | 10% | 고객사 지불 능력, 의사결정 구조, 과거 협업 이력 |

> **참고:** 가중치는 초기 기본값이며, 관리자가 설정 화면에서 조정 가능하도록 구현한다.

3.3 OathKeeper 에이전트가 예상 참여인력(PM, Front/Backend, MLE 등), 기간, 소요 예산 등을 계산. 이때 사전 준비 단계에서 설정한 회사 사업방향, 전략, 규정 문서, 그리고 과거 수행사례도 함께 고려함.

3.4 Oathkeeper 에이전트가 deal의 잠재적 리스크 요인을 카테고리별로 식별하고, 심각도에 따른 경고를 제시

- 리스크 카테고리
	- **기술 리스크:** 미경험 기술, 높은 구현 복잡도, 데이터 품질 이슈 등
	- **일정 리스크:** 촉박한 납기, 불명확한 마일스톤, 병렬 작업 불가 등
	- **재무 리스크:** 낮은 마진, 불명확한 결제 조건, 추가 비용 발생 가능성 등
	- **고객 리스크:** 의사결정 지연 가능성, 요구사항 변경 빈도, 지불 안정성 등
	- **범위 리스크:** Scope creep 가능성, 모호한 성공 기준, 암묵적 추가 요구 등

- 경고 레벨: 각 리스크 항목을 🔴 **높음** (미해결 시 프로젝트 실패 가능성) / 🟡 **중간** (추가 비용 발생 가능성) / 🟢 **낮음** (관리 가능한 수준)으로 분류하여 표시한다.

3.5 Oathkeekper 에이전트가 과거 수행한 프로젝트 이력을 기반으로 유사한 사례를 검색하고, 해당 사례의 결과와 교훈을 함께 제공.
- 기능 상세
- **유사성 기준:** 산업군, 기술 스택, 프로젝트 규모, 고객 유형 등을 기반으로 유사도 점수 산출
- **비교 정보:** 유사 프로젝트의 실제 투입 리소스, 발생 이슈, 최종 결과(성공/실패/초과) 제공
- **Lessons Learned:** 유사 프로젝트에서 발생한 이슈와 그 해결 방법을 참고 정보로 제공

3.6. Oathkeeper 에이전트가 작성한 결과를 바탕으로 Go/ No GO 여부를 판단.
**종합 판단 로직**

| 판단 | 조건 | 설명 |
|---|---|---|
| ✅ **Go** | 종합 70점 이상 | 수행 권고. 주요 리스크 요인 함께 제시 |
| ⚠️ **조건부 Go** | 종합 40~69점 | 조건 충족 시 수행. 해소 필요 항목 명시 |
| ❌ **No-Go** | 종합 40점 미만 | 수행 비권고. 주요 사유 및 대안 제시 |
| ℹ️ **보류** | 필수 평가 항목 중 미입력 존재 | 판단 보류. 추가 확인이 필요한 항목을 구체적으로 안내 |

4. 분석한 내용을 정리하여 화면에 출력.
5. 사용자가 허가(수정, 종료, 저장)를 받아 사용자가 저장 요청하면 노션의 해당 deal, 분석 결과 페이지에 내용을 저장.

## 기술 스택 (고정)

| 구분 | 기술 | 비고 |
|---|---|---|
| LLM | OpenAI GPT-4o / Claude Sonnet | 용도별 최적 모델 선택 (A/B 테스트) |
| Backend | Python (FastAPI) | LLM 연동 및 Agent 오케스트레이션 |
| Agent | Langchain, Langgraph | RAG, Agent 워크플로우 설계 |
| Vector DB | Pinecone | 과거 프로젝트 유사도 검색용 |
| RDB | PostgreSQL | 평가 결과, 설정, 사용자 데이터 저장 |
| Frontend | Nextjs, TailwindCSS v4 + shadcn/ui | 사용자 입출력, 분석 결과, 관리화면 등 |
| Input 연동 | Notion API | MVP 기준. 향후 HubSpot API 추가 |
| 알림 | Slack Webhook | 분석 완료 알림 및 요약 전송 |

## PRD 작성 요구사항

### 1. 문서 구조
다음 섹션을 포함하여 작성:

1. **프로젝트 개요**
   - 프로젝트 이름 및 한줄 설명
   - 해결하려는 문제
   - 핵심 가치 제안

2. **사용자 정의**
   - 주요 사용자 (관리자/클라이언트)
   - 각 사용자별 니즈와 목표

3. **핵심 기능 (MVP 범위)**
   - 필수 기능 목록 (Must Have)
   - 우선순위 및 의존성
   - 각 기능별 상세 요구사항

4. **시스템 아키텍처**
   - 데이터 흐름도
   - 에이전트 구성 및 워크 플로우
   - 페이지 라우팅 구조

5. **데이터 모델**
   - 전체 데이터(RDB, Vector DB) 구성
   - RDB 스키마
   - Vector DB 스키마
   - 노션 데이터베이스 필드 매핑

6. **UI/UX 요구사항**
   - 페이지별 와이어프레임 설명
   - 반응형 디자인 요구사항
   - 사용자 입력 페이지 요구사항
   - 최대 4개 페이지 이하로 구성

7. **프롬프트 관리/ 구성**

8. **MVP 제외 사항**
   - 향후 버전에서 고려할 기능
   - 의도적으로 제외한 기능과 이유

9. **성공 지표**
   - MVP 완료 기준
   - 측정 가능한 목표

### 2. 작성 원칙

- **구체적으로**: 모호한 표현 대신 명확한 요구사항 기술
- **개발자 친화적**: 바로 구현 가능한 수준의 상세도
- **MVP 집중**: 최소 기능에 집중, 과도한 기능 배제
- **실용적**: 1인 개발자가 2-4주 내 구현 가능한 범위
```
